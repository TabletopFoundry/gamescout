# Troubleshooting & FAQ

## Install & build

### `Error: The module 'better_sqlite3.node' was compiled against a different Node.js version`

You upgraded Node (or switched versions) after running `npm install`. Rebuild the native binding:

```bash
npm rebuild better-sqlite3
```

### `npm install` fails compiling `better-sqlite3`

You're missing a C toolchain.

- **macOS:** `xcode-select --install`
- **Debian/Ubuntu:** `sudo apt install build-essential python3`
- **Windows:** install Visual Studio Build Tools with the C++ workload

Then re-run `npm install`.

### Port 3000 already in use

```bash
PORT=3001 npm run dev
```

### Stale type errors after pulling

```bash
rm -rf .next
npm run dev
```

## Database

### "no such table: games" on first request

The seed didn't run. Check:

- You're running `npm run dev` (or `npm run start` with `GAMESCOUT_ALLOW_PRODUCTION_SEED=1`)
- The working directory is writable
- The repo has the `src/data/seed/` directory intact

Delete the database files and retry:

```bash
rm gamescout.db gamescout.db-shm gamescout.db-wal
npm run dev
```

### How do I reset to a clean seeded state?

```bash
rm gamescout.db gamescout.db-shm gamescout.db-wal
npm run dev
```

The next request reseeds.

### How do I edit the seeded data?

Edit `src/data/seed/games.ts`, `personas.ts`, or `deals.ts`, then bump `SEED_VERSION` in `src/lib/seed.ts`. The next boot will detect the new version and re-run the seeder.

### Will the seed run in production?

No — not unless you set `GAMESCOUT_ALLOW_PRODUCTION_SEED=1`. The guard exists so a routine production redeploy doesn't wipe your data.

## Sessions

### Why am I being asked to take the quiz again?

You cleared cookies, switched browsers, or used incognito mode. Sessions are tied to the `gs_session` cookie. See [Sessions & Users](./concepts/sessions-and-users).

### Can I share a session between two devices?

Not out of the box. Copy the `gs_session` cookie value from one device to the other manually, or [add real auth](./concepts/sessions-and-users#adding-real-auth-later).

## Recommendations

### Why are my recommendations the same as the BGG top 100?

You haven't taken the quiz yet, or you've only marked games as "haven't played". The engine falls back to BGG rating when it has no positive signal. Take the quiz and rate at least three games as `loved` or `liked`.

### The "reason" string is wrong

The reason is generated by the same arithmetic that computes the score, so it can't be wrong in the way you'd expect — but it can be *uninformative* when the top contributor is a small term. Open `src/lib/recommendations.ts` and adjust the threshold above which a fragment is added to the reason list.

## Prices & Alerts

### My price alert was never delivered

GameScout doesn't include an email sender in the open-source build. You'll see the alert recorded in `price_alerts` with `active = 1`, but you need to wire up your own delivery cron. See [Price Tracking](./guides/price-tracking#a-note-on-delivery).

### Prices in the demo are stale

They are — seed data is checked in at a moment in time. Refresh by replacing rows in `price_history`. The schema is documented in [Database Schema](./reference/database-schema).

## Deployment

### My data disappears between requests in production

You're deploying to an ephemeral filesystem (some serverless platforms). SQLite needs persistent disk. See [Deploying GameScout](./guides/deploying#persistent-storage-gotchas).

### How do I back up?

Either copy `gamescout.db` while the server is idle, or use SQLite's online backup:

```bash
sqlite3 gamescout.db ".backup /backups/gamescout-$(date +%F).db"
```

WAL mode is on, so even live copies of the file are safe.
